2. Mutation path (write a row)

Every write is optimistic locally, queued in the outbox if offline, and reconciled via Realtime echoes.

sequenceDiagram autonumber participant U as User participant IL as ItemList.tsx participant M as sync/mutations.ts participant DX as Dexie (local) participant OB as Outbox participant E as sync/engine.ts participant SA as Server Action participant PG as Postgres (RLS) participant RT as Realtime participant RC as sync/reconcile.ts U->>IL: tap "add item" IL->>IL: optimistic insert into local state IL->>M: mutate(addItem) M->>DX: write local row M->>OB: enqueue mutation par offline-safe flush E->>OB: drain queue E->>SA: call server action SA->>PG: insert / update (RLS-checked) PG-->>SA: ok / no rows SA-->>E: { error? } E->>OB: ack or retry and realtime echo PG->>RT: change feed RT-->>RC: INSERT/UPDATE/DELETE RC->>DX: merge into cache RC-->>IL: state update end alt server returned error SA-->>IL: { error } IL->>IL: rollback from snapshot end

4. Data model (Supabase)

Authorization lives in Postgres via RLS using has_list_access(uuid); the app does not re-check.

Notes

  • Three Supabase clients — server.ts (Server Components/Actions), client.ts (browser, Realtime), middleware.ts (edge). Don't mix them.
  • Edge middleware = src/proxy.ts — Next 16 rename. Excludes /manifest.webmanifest and /sw.js so Chrome's PWA installability check passes.
  • Optimistic UI lives in ItemList.tsx; every handler snapshots, mutates locally, awaits the server action, rolls back on error, and reconciles with the Realtime echo.
  • Offline cache — Dexie/IndexedDB holds lists and items; mutations queue through an outbox flushed by sync/engine.ts; reconcile.ts merges Realtime events.
  • Categories — 11-slug closed enum in lib/categories.ts; first hit via user_item_history, fallback to Gemini, manual override in edit modal.
