Inside a task list — UI exploration

Four takes on the task detail view, on the real list-detail chrome (back arrow, title, add field, rounded rows, struck-through done section). They escalate left→right, top→bottom: from a bare shared checklist to assignment, due dates, and grouping. Pick how much machinery is worth it for "family tasks".

A — bare shared checklist

minimal
Shopping's model with an explicit checkbox instead of tap-to-check. No assignment, no dates. Anyone checks anything; done drops to the bottom struck through. Almost zero new concepts.

B — assignee + optional due date

assignment
The real "family" value: each task can carry an assignee avatar and a due-date pill (amber = soon, red = overdue, grey = future). Both optional — unassigned shows a hollow avatar. Sorted by due date.

D — rich rows: note + priority

heaviest
Each task can hold a one-line note/sub-detail and a priority spine (red/amber/grey on the left edge). Most expressive, closest to a real task manager — and the most scope-creep risk. Shown as the "if we go big" end of the range.

Reading the range

  • A ships in days — it's the shopping list minus groceries plus a checkbox. Validates "do we even use this?" cheaply.
  • B is the sweet spot for "family tasks": assignment + due answers who and when without a real scheduler.
  • C is B's data re-grouped — a view toggle, not new fields. Could be a sort option on top of B rather than its own build.
  • D opens the door to notes/priority/sub-tasks — powerful but this is where "todo app" scope-creep starts.

Cost vs. payoff

  • A reuses items table as-is (name + is_checked). New columns only from B onward (assignee_id, due_date).
  • B/C share one schema; C is purely a client-side grouping of B.
  • Due dates imply expectations: sorting, "overdue" styling, maybe reminders. Decide if you're OK stopping at visual-only.
  • D's note/priority touch the edit modal, add-flow, and sorting — biggest surface, least certain payoff for a family chore list.

My lean: build B as the target, ship A first if you want to de-risk usage. Treat C as a later sort/group toggle over B's data, and hold D unless the family explicitly wants notes. Due dates = visual + sort only for v1 — no reminders/notifications.
